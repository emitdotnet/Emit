## IEventProducer

Once a topic has been declared with `topic.Producer()`, Emit registers `IEventProducer<TKey, TValue>` as a singleton in the DI container. Inject it wherever you need to produce messages:

```csharp
public class PizzeriaService(IEventProducer<string, PizzaOrdered> producer)
{
    public async Task PublishAsync(PizzaOrdered pizza, CancellationToken ct)
    {
        await producer.ProduceAsync(
            new EventMessage<string, PizzaOrdered>(pizza.PizzaId.ToString(), pizza),
            ct);
    }
}
```

`IEventProducer` is a singleton. Inject it freely into scoped or transient services; the DI container handles the lifetime mismatch correctly.

## EventMessage

`EventMessage<TKey, TValue>` carries the key, value, and optional headers:

```csharp
// Key + value only
var msg = new EventMessage<string, PizzaOrdered>(pizza.PizzaId.ToString(), pizza);

// With headers
var msg = new EventMessage<string, PizzaOrdered>(
    pizza.PizzaId.ToString(),
    pizza,
    [new KeyValuePair<string, string>("x-tenant-id", tenantId)]);
```

Headers are passed through to Kafka as-is. Emit also appends its own tracing headers when OpenTelemetry is enabled.

## Outbox mode vs. direct mode

Whether the message goes through the outbox or straight to Kafka depends on whether you called `UseOutbox()`:

**With outbox** (`UseOutbox()` called): `ProduceAsync` writes to the database outbox inside the current `IEmitContext.Transaction`. The background worker delivers to Kafka later. Your calling code must begin a transaction first.

**Without outbox**: `ProduceAsync` calls the Kafka producer directly and waits for the broker acknowledgement before returning.

Use the outbox when you need atomicity with your database writes. Use direct mode when you want simplicity and can tolerate the dual-write risk or don't have a database write to coordinate with.

## Per-producer middleware

Register middleware that runs only for a specific topic's producer:

```csharp
kafka.Topic<string, PizzaOrdered>("pizzas", topic =>
{
    topic.Producer(producer =>
    {
        producer.Use<EnrichmentMiddleware>();
    });
});
```

The middleware receives `OutboundContext<PizzaOrdered>` and calls `next` to continue the pipeline. This is the Leaf level of the [pipeline model](../concepts/pipeline-model/).

## Accessing the message key in middleware

The Kafka key is available via `IKeyFeature<TKey>` on the outbound context:

```csharp
public class LoggingMiddleware<T> : IMiddleware<OutboundContext<T>>
{
    public Task InvokeAsync(OutboundContext<T> context, MessageDelegate<OutboundContext<T>> next)
    {
        if (context.Features.Get<IKeyFeature<string>>() is { } keyFeature)
        {
            logger.LogDebug("Producing with key {Key}", keyFeature.Key);
        }
        return next(context);
    }
}
```

## Produce observers

`IProduceObserver` gives lifecycle hooks without writing middleware:

```csharp
public class ProduceAuditObserver : IProduceObserver
{
    public Task OnProducedAsync<T>(OutboundContext<T> context) { ... }
    public Task OnProduceErrorAsync<T>(OutboundContext<T> context, Exception exception) { ... }
}
```

Register at the global level on `EmitBuilder`:

```csharp
emit.AddProduceObserver<ProduceAuditObserver>();
```
