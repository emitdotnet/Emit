## IConsumer

`IConsumer<TValue>` is where your message handling lives. Implement it, register it on a consumer group, and Emit takes care of the rest:

```csharp
public class PizzaOrderedConsumer(ILogger<PizzaOrderedConsumer> logger)
    : IConsumer<PizzaOrdered>
{
    public async Task ConsumeAsync(ConsumeContext<PizzaOrdered> context, CancellationToken cancellationToken)
    {
        var pizza = context.Message;
        logger.LogInformation("Preparing pizza {PizzaId} for {CustomerId}", pizza.PizzaId, pizza.CustomerId);

        // route to kitchen, notify delivery, whatever your domain needs
    }
}
```

Register it on the consumer group:

```csharp
kafka.Topic<string, PizzaOrdered>("pizzas", topic =>
{
    topic.ConsumerGroup("pizza-kitchen", group =>
    {
        group.AddConsumer<PizzaOrderedConsumer>();
    });
});
```

Consumers are resolved from the DI container per message, so constructor injection works normally.

## ConsumeContext

`ConsumeContext<TValue>` carries the message and metadata:

| Member | Type | Description |
|---|---|---|
| `Message` | `TValue` | The deserialized message value |
| `MessageId` | `string` | Unique message identifier |
| `Timestamp` | `DateTimeOffset` | Message timestamp |
| `DestinationAddress` | `Uri?` | Transport URI (e.g. `kafka://broker:9092/orders`) |
| `SourceAddress` | `Uri?` | Sender address, propagated via headers |
| `Headers` | `IReadOnlyList<...>` | Message headers (delegated from transport context) |
| `RetryAttempt` | `int` | Current retry count (0 on first attempt) |
| `CancellationToken` | `CancellationToken` | Cancellation for this message |
| `Services` | `IServiceProvider` | Scoped provider for this message |
| `TransportContext` | `TransportContext` | Transport-specific metadata |
| `Transaction` | `ITransactionContext?` | Active transaction, if any |

### Accessing Kafka-specific data

The `TransportContext` is a `KafkaTransportContext<TKey>` on the consume side. Cast it to access the deserialized message key and Kafka metadata:

```csharp
if (context.TransportContext is KafkaTransportContext<string> kafka)
{
    logger.LogInformation("Key={Key} Topic={Topic} Partition={Partition} Offset={Offset}",
        kafka.Key, kafka.Topic, kafka.Partition, kafka.Offset);
}
```

See [Feature Collection & Payload Bag](../../advanced/feature-collection/) for more on context properties and payload access.

## Worker pool

The worker pool controls how many messages your instance processes at the same time. To size it correctly, you need to understand how Kafka distributes work — starting with partitions.

### Partitions: the unit of parallelism

A Kafka topic is not a single queue. It is divided into **partitions** — independent, ordered logs. Inside a single partition, messages are always delivered in the order they were written. Across different partitions, there is no ordering guarantee whatsoever.

When a producer sends a message to `orders`, Kafka assigns it to one partition — typically based on the message key. The more partitions a topic has, the more consumers can read from it simultaneously.


### Consumer groups and partition assignment

A **consumer group** is a named set of consumers that share the work of reading a topic. Kafka's contract: each partition is consumed by **exactly one member** of the group at a time. Two consumers in the same group never read the same partition simultaneously.

When you run multiple instances of your application with the same group ID, Kafka divides the partitions between them automatically. This redistribution — called a **rebalance** — happens whenever an instance joins or leaves the group. No configuration needed; Kafka handles it.

<Mermaid code={`flowchart LR
    subgraph i1[Instance 1  ·  pizza-kitchen]
        p0[Partition 0]
        p1[Partition 1]
        p2[Partition 2]
    end
    subgraph i2[Instance 2  ·  pizza-kitchen]
        p3[Partition 3]
        p4[Partition 4]
        p5[Partition 5]
    end`} />

Scale to three instances and each gets two partitions. Take one down and Kafka reassigns its partitions to the survivors within seconds. Horizontal scaling with zero code changes.

### Workers within an instance

Each application instance runs one Kafka consumer per group. That consumer polls its assigned partitions and feeds messages into Emit's **worker pool** — a set of concurrent processing channels within the instance.

Each worker handles one message at a time. With three workers, the instance processes up to three messages simultaneously. A fourth message waits for a slot to open up, or queues in the channel buffer (up to `BufferSize`) while it waits.

<Mermaid code={`flowchart LR
    subgraph kafka[Kafka]
        p0[Partition 0]
        p1[Partition 1]
        p2[Partition 2]
    end
    subgraph pool[Worker pool]
        w1[Worker 1]
        w2[Worker 2]
        w3[Worker 3]
    end
    p0 --> w1
    p1 --> w2
    p2 --> w3`} />

```csharp
topic.ConsumerGroup("pizza-kitchen", group =>
{
    group.WorkerCount = 3;
    group.BufferSize = 32;  // per-worker channel capacity

    group.AddConsumer<PizzaOrderedConsumer>();
});
```

### How many workers?

**The rule of thumb: match `WorkerCount` to the number of partitions your instance will be assigned.**

If `orders` has 6 partitions and you run 2 instances, Kafka assigns 3 partitions to each. Set `WorkerCount = 3` and each partition gets a dedicated worker — full parallelism, no contention.

**More workers than assigned partitions?** Extra workers stay idle. They hold a small buffer channel in memory but cause no harm. If you scale dynamically and can't predict the exact partition count per instance, erring slightly high is safe.

**Fewer workers than assigned partitions?** Multiple partitions compete for the same workers. A slow message from Partition 0 can hold up messages from Partition 1 and Partition 2, even though those messages are already sitting in the buffer ready to go. You lose the parallelism the partitions were designed to give you.

## Distribution strategies

The distribution strategy answers one question: when a message arrives, which worker gets it? It only matters when `WorkerCount > 1`.

### ByKeyHash (default)

Emit computes `hash(key) % WorkerCount` and routes the message to the resulting worker. Because the hash of a key never changes, **the same key always maps to the same worker** — for every message, without exception.

Think of it as giving each customer a dedicated lane. Customer 42's orders always land on Worker 2. Customer 99's orders always land on Worker 3. If three orders arrive for Customer 42 in quick succession, Worker 2 processes them in sequence. Worker 3 is busy with Customer 99's orders in parallel — neither lane waits for the other.

<Mermaid code={`flowchart LR
    subgraph g42[cust-42]
        o1[order-1]
        o3[order-3]
    end
    subgraph g99[cust-99]
        o2[order-2]
    end
    subgraph g15[cust-15]
        o4[order-4]
    end
    w1([Worker 1])
    w2([Worker 2])
    w3([Worker 3])
    g42 --> w2
    g99 --> w3
    g15 --> w1`} />

`order-1` and `order-3` both belong to `cust-42`, so both route to Worker 2 and process in arrival order. `order-2` (cust-99) runs on Worker 3 at the same time — the two customers are completely independent of each other.

The tradeoff: load isn't perfectly balanced. A very active key monopolizes its worker while others run lighter. That's usually the right tradeoff — correctness is worth more than perfect load distribution.

Use `ByKeyHash` when ordering within a key matters: state transitions for the same entity, inventory updates for the same product SKU, anything where two messages about the same subject must not process out of order.

### RoundRobin

Workers are assigned in rotation — message 1 goes to Worker 0, message 2 to Worker 1, message 3 to Worker 2, then back to Worker 0. The message key is ignored entirely.

<Mermaid code={`flowchart LR
    e1[event-1] --> w0([Worker 0])
    e2[event-2] --> w1([Worker 1])
    e3[event-3] --> w2([Worker 2])
    e4[event-4] --> w3([Worker 3])
    e5[event-5] --> w0`} />

This spreads load as evenly as possible and maximizes throughput across the pool. The cost is that two messages from the same source can land on different workers and complete in a different order than they arrived.

Use `RoundRobin` when messages are independent and throughput matters more than ordering: audit events, telemetry records, fire-and-forget notifications — anything where it's fine if two events from the same customer finish in a different order than they were sent.

## Offset commit

Emit commits offsets automatically using a watermark algorithm: the offset advances only when all messages up to that point have been processed. A slow message at offset 100 blocks the commit at 99 even if 101 through 200 have already finished.

Configure the commit interval:

```csharp
group.CommitInterval = TimeSpan.FromSeconds(5);  // default
```

Commits happen in the background on this interval. You do not call commit manually.

## Per-consumer middleware

Register middleware that wraps only a specific consumer:

```csharp
group.AddConsumer<PizzaOrderedConsumer>(consumer =>
{
    consumer.Use<RetryLoggingMiddleware>();
});
```

This is the innermost middleware layer for that consumer, running inside any group-level middleware.

## Consumer group middleware

Middleware registered on the group itself wraps all consumers in that group:

```csharp
topic.ConsumerGroup("pizza-kitchen", group =>
{
    group.Use<TracingMiddleware>();

    group.AddConsumer<PizzaOrderedConsumer>();
});
```

For error handling, validation, circuit breakers, and rate limiting at the group level, see [Error Policies](../../error-handling/error-policies/) and [Resilience](../../error-handling/resilience/).

## Graceful shutdown

When the application stops, Emit waits up to `WorkerStopTimeout` (default 30 seconds) for in-flight messages to complete before forcing shutdown:

```csharp
group.WorkerStopTimeout = TimeSpan.FromSeconds(30);
```

## Message filtering

`IConsumerFilter<TMessage>` lets you skip messages before they reach the handler. Return `false` to short-circuit the pipeline without triggering the error policy — the offset is committed and the message is silently dropped.

```csharp
public class ActiveOrderFilter : IConsumerFilter<PizzaOrdered>
{
    public ValueTask<bool> ShouldConsumeAsync(ConsumeContext<PizzaOrdered> context, CancellationToken ct)
        => ValueTask.FromResult(context.Message.Status == OrderStatus.Active);
}
```

Register it on the consumer:

```csharp
group.AddConsumer<PizzaOrderedConsumer>(consumer =>
{
    consumer.Filter<ActiveOrderFilter>();
});
```

For inline predicates without a dedicated class:

```csharp
// Synchronous
consumer.Filter<PizzaOrdered>(ctx => ctx.Message.Status == OrderStatus.Active);

// Async
consumer.Filter<PizzaOrdered>(async (ctx, ct) =>
    await featureFlags.IsEnabledAsync("pizza-kitchen", ct));
```

## Consumer observers

`IKafkaConsumerObserver` provides Kafka-specific lifecycle events. Register it on the Kafka builder:

```csharp
kafka.AddConsumerObserver<MyKafkaObserver>();
```

All methods have default no-op implementations — implement only the ones you care about:

| Method | Event record | When it fires |
|---|---|---|
| `OnConsumerStartedAsync` | `ConsumerStartedEvent(GroupId, Topic, WorkerCount)` | Worker begins polling |
| `OnConsumerStoppedAsync` | `ConsumerStoppedEvent(GroupId, Topic)` | Worker stops cleanly |
| `OnConsumerFaultedAsync` | `ConsumerFaultedEvent(GroupId, Topic, Exception)` | Worker crashes and triggers a restart |
| `OnPartitionsAssignedAsync` | `PartitionsAssignedEvent(GroupId, Topic, Partitions)` | Rebalance assigns new partitions |
| `OnPartitionsRevokedAsync` | `PartitionsRevokedEvent(GroupId, Topic, Partitions)` | Rebalance gracefully revokes partitions |
| `OnPartitionsLostAsync` | `PartitionsLostEvent(GroupId, Topic, Partitions)` | Unclean partition loss |
| `OnOffsetsCommittedAsync` | `OffsetsCommittedEvent(GroupId, Offsets)` | Offsets successfully committed |
| `OnOffsetCommitErrorAsync` | `OffsetCommitErrorEvent(GroupId, Offsets, Exception)` | Offset commit fails |
| `OnDeserializationErrorAsync` | `DeserializationErrorEvent(GroupId, Topic, Partition, Offset, Exception)` | Message fails deserialization before pipeline entry |

`OnConsumerFaultedAsync` and `OnPartitionsLostAsync` are the most operationally important — they signal conditions that require attention rather than normal lifecycle transitions.

For transport-agnostic consume lifecycle hooks, use `IConsumeObserver` at the global level:

```csharp
emit.AddConsumeObserver<MyConsumeObserver>();
```

See [Observers](../../observability/observers/) for details on both.
