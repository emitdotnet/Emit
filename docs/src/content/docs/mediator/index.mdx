The mediator dispatches typed requests to their registered handlers in-process. It uses the same middleware pipeline model as everything else in Emit, which means tracing, metrics, observers, and custom middleware all work without any extra wiring.

## Registration

```csharp title="Program.cs"
builder.Services.AddEmit(emit =>
{
    emit.AddMediator(mediator =>
    {
        mediator.AddHandler<PlacePizzaOrderHandler>();
        mediator.AddHandler<GetPizzaStatusHandler>();
    });
});
```

`AddMediator` is an extension on `EmitBuilder`. You can call it alongside `AddKafka` or on its own; the two are independent.

## Requests

A request is a plain class or record that implements `IRequest` (no response) or `IRequest<TResponse>` (returns a value):

```csharp
// Fire-and-forget
public record PlacePizzaOrder(string CustomerId, string[] Toppings, string Address) : IRequest;

// Returns a result
public record GetPizzaStatus(Guid PizzaId) : IRequest<PizzaStatusDto>;
```

No base types, no attributes, no conventions to memorize.

## Handlers

Implement `IRequestHandler<TRequest>` for void handlers, or `IRequestHandler<TRequest, TResponse>` for handlers that return a value:

```csharp
public class PlacePizzaOrderHandler(IPizzaRepository pizzas) : IRequestHandler<PlacePizzaOrder>
{
    public async Task HandleAsync(PlacePizzaOrder request, CancellationToken cancellationToken)
    {
        await pizzas.SaveAsync(new PizzaOrder(Guid.NewGuid(), request.CustomerId, request.Toppings, request.Address), cancellationToken);
    }
}

public class GetPizzaStatusHandler(IPizzaRepository pizzas) : IRequestHandler<GetPizzaStatus, PizzaStatusDto>
{
    public async Task<PizzaStatusDto> HandleAsync(GetPizzaStatus request, CancellationToken cancellationToken)
    {
        var order = await pizzas.FindAsync(request.PizzaId, cancellationToken);
        return new PizzaStatusDto(order.PizzaId, order.Status, order.EstimatedDelivery);
    }
}
```

Handlers are resolved from the DI container, so constructor injection works normally.

## Dispatching

Inject `IMediator` wherever you need it:

```csharp
public class PizzeriaService(IMediator mediator)
{
    public Task PlaceAsync(PlacePizzaOrderRequest req, CancellationToken ct)
        => mediator.SendAsync(new PlacePizzaOrder(req.CustomerId, req.Toppings, req.Address), ct);

    public Task<PizzaStatusDto> GetStatusAsync(Guid pizzaId, CancellationToken ct)
        => mediator.SendAsync<PizzaStatusDto>(new GetPizzaStatus(pizzaId), ct);
}
```

Sending to an unregistered request type throws `InvalidOperationException` at dispatch time.

## Pipeline

Middleware runs at three levels before the handler is invoked:

Register mediator-level middleware on `mediator.InboundPipeline`:

```csharp
emit.AddMediator(mediator =>
{
    mediator.InboundPipeline.Use<LoggingMiddleware>();

    mediator.AddHandler<PlacePizzaOrderHandler>();
});
```

Register per-handler middleware using the two-argument overload:

```csharp
mediator.AddHandler<PlacePizzaOrderHandler, PlacePizzaOrder>(handler =>
{
    handler.Use<ValidationMiddleware>();
});
```

The context flowing through all middleware layers is `MediatorContext<TRequest>`. The terminal adapter extracts `context.Message` and calls `HandleAsync`.

## Observers

`IMediatorObserver` gives you lifecycle hooks without writing middleware:

```csharp
public class AuditObserver : IMediatorObserver
{
    public Task OnHandlingAsync<T>(MediatorContext<T> context)
    {
        // called before the handler
        return Task.CompletedTask;
    }

    public Task OnHandledAsync<T>(MediatorContext<T> context)
    {
        // called after successful completion
        return Task.CompletedTask;
    }

    public Task OnHandleErrorAsync<T>(MediatorContext<T> context, Exception exception)
    {
        // called on exception
        return Task.CompletedTask;
    }
}
```

Register it on the mediator builder:

```csharp
mediator.AddObserver<AuditObserver>();
```

Observer exceptions are swallowed and logged; a failing observer never interrupts the handler.

## One handler per request type

Registering two handlers for the same request type throws at startup. Fan-out belongs inside a single handler: dispatch to multiple targets explicitly, or publish an event through the outbox.
