Every message in Emit passes through a layered middleware pipeline before reaching its handler or destination. The inbound path is richer than it looks: a raw Kafka message goes through deserialization and fan-out before individual consumer pipelines even start.

## Inbound pipeline (consuming)

The inbound path has two distinct phases: **transport processing** (deserialization and fan-out) followed by **per-consumer pipeline execution**.

### Transport phase

When a message arrives from Kafka, the `ConsumerWorker` handles deserialization and fan-out *before* any middleware runs:

<Mermaid code={`flowchart TB
    K[["Kafka broker"]]
    W["ConsumerWorker"]
    D["Deserialize key + value"]
    FO{"Fan-out"}
    C1["Consumer A pipeline"]
    C2["Consumer B pipeline"]
    C3["Router pipeline"]
    K --> W --> D --> FO
    FO --> C1
    FO --> C2
    FO --> C3`} />

1. **Poll**: The consumer group worker reads raw bytes from Kafka and distributes them to worker channels
2. **Deserialize**: Key and value are deserialized using the registered `IDeserializer<T>` / `IAsyncDeserializer<T>`. Deserialization errors are handled separately (dead-letter or discard) â€” they never enter the consumer pipeline
3. **Fan-out**: For each consumer or router registered on this consumer group, a fresh DI scope is created along with its own `KafkaTransportContext<TKey>` and `ConsumeContext<TValue>`. Each entry runs its pipeline independently

Deserialization errors are handled via `OnDeserializationError` on the consumer group. Since the message hasn't been deserialized yet, these errors bypass the consumer pipeline entirely.

### Consumer pipeline

The exact layer order (outermost to innermost):

| Layer | What it does | Always present? |
|---|---|---|
| **Error handling** | Catches exceptions, evaluates error policy (dead-letter, discard), notifies circuit breaker | Yes |
| **Observers** | Invokes registered `IConsumeObserver` callbacks | Yes |
| **Tracing** | Creates `emit.consume` Activity with consumer identity | Yes |
| **Metrics** | Records duration, success/failure counters | Yes |
| **User middleware** | Global â†’ Provider â†’ Group middleware, in registration order | Only if registered |
| **Validation** | Runs `IMessageValidator<T>` before the handler; failures skip retry | Only if configured |
| **Retry** | Re-executes handler + per-entry middleware on failure with backoff | Only if configured |
| **Per-consumer middleware** | Middleware registered on `AddConsumer` or per-route | Only if registered |
| **Handler** | `IConsumer<T>.ConsumeAsync()` | Yes |

Error handling is the outermost layer so it catches exceptions from every other layer. Validation runs outside retry so validation failures are never retried. Tracing and metrics are baked with the consumer's identity at build time.

### Fan-out and routing

When a consumer group has multiple consumers or routers, each one gets its own pipeline execution per message. They share the same deserialized message but each gets:

- A fresh `IServiceScope`
- Its own `KafkaTransportContext<TKey>` and `ConsumeContext<TValue>` instances
- Independent pipeline execution (one failing doesn't affect others)

**Content-based routing** adds a second level of dispatch. A `MessageRouterInvoker` sits at the terminal position of the consumer pipeline. It evaluates the route selector, then invokes the matched route's sub-pipeline:

## Context objects

Middleware receives a typed context that determines which pipeline direction it belongs to:

- **`ConsumeContext<T>`** â€” inbound consumer pipeline. Carries the deserialized message and a reference to its parent `TransportContext`
- **`SendContext<T>`** â€” outbound producer pipeline. Carries the message and mutable headers

Both extend `MessageContext<T>` â†’ `MessageContext`, which provides common properties: `MessageId`, `Timestamp`, `DestinationAddress`, `SourceAddress`, `Services`, a feature collection, and a payload bag.

### TransportContext

`ConsumeContext<T>` holds a reference to its parent `TransportContext`, which carries pre-deserialization data: raw bytes, headers, and the provider ID. For Kafka, this is `KafkaTransportContext<TKey>` with topic, partition, offset, group ID, and the deserialized key:

```csharp
if (context.TransportContext is KafkaTransportContext<string> kafka)
    logger.LogDebug("Key: {Key}, Partition: {P}", kafka.Key, kafka.Partition);
```

For outbound messages, `KafkaTransportContext<TKey>` is stored as a payload on the `SendContext` (via `SetPayload`) so the terminal can extract the key for serialization.

See [Feature Collection & Payload Bag](../../advanced/feature-collection/) for the full list of context properties and built-in payloads.

## Middleware levels

User middleware is composed across three levels, each narrowing scope:

| Level | Scope | Configured on |
|---|---|---|
| **Global** | Every message, every provider | `EmitBuilder` |
| **Provider** | All messages for one provider | `KafkaBuilder` |
| **Leaf** | One consumer group, producer, or handler | Group/producer/handler builder |

Middleware registered at a higher level runs *outside* middleware at a lower level, just like ASP.NET middleware wraps your endpoint. Within each level, middleware executes in registration order.

## Middleware lifetimes

```csharp
group.Use<MyMiddleware>(MiddlewareLifetime.Singleton);  // one shared instance (default)
group.Use<MyMiddleware>(MiddlewareLifetime.Scoped);      // one instance per message
```

`Singleton` is the default: one instance shared across all concurrent workers. It must be thread-safe and cannot hold per-message state. `Scoped` middleware gets a fresh instance for each message and can safely hold per-message state.

## What's registered automatically

Emit always registers error handling, tracing, metrics, and observer middleware. You don't add them manually. The order is fixed: error handling (outermost) â†’ observers â†’ tracing â†’ metrics â†’ your middleware â†’ validation â†’ retry â†’ handler (innermost). Your custom middleware runs inside the built-in layers.

For more on writing middleware, see [Custom Middleware](../../advanced/custom-middleware/).
