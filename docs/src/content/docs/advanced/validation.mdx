---
title: Validation
description: Validate messages before they reach your consumer using built-in validators or FluentValidation.
---

Emit validates messages after deserialization and before they reach your consumer handler. Invalid messages are either discarded or dead-lettered, depending on the error action you configure.

## Delegate Validators

The simplest option: pass a function directly.

```csharp
group.Validate(
    msg => msg.Amount > 0
        ? MessageValidationResult.Success
        : MessageValidationResult.Fail("Amount must be positive"),
    onFailure => onFailure.Discard());
```

Async delegates are also supported:

```csharp
group.Validate(
    async (msg, cancellationToken) =>
    {
        var exists = await db.ExistsAsync(msg.Id, cancellationToken);
        return exists
            ? MessageValidationResult.Success
            : MessageValidationResult.Fail("Entity not found");
    },
    onFailure => onFailure.DeadLetter());
```

## Class-Based Validators

Implement `IMessageValidator<T>` for reusable validation logic:

```csharp
public class OrderValidator : IMessageValidator<OrderCreated>
{
    public Task<MessageValidationResult> ValidateAsync(
        OrderCreated message, CancellationToken cancellationToken)
    {
        if (string.IsNullOrWhiteSpace(message.CustomerId))
            return Task.FromResult(MessageValidationResult.Fail("CustomerId is required"));

        return Task.FromResult(MessageValidationResult.Success);
    }
}
```

Register it on the consumer group:

```csharp
group.Validate<OrderValidator>(onFailure => onFailure.Discard());
```

The validator is resolved from DI automatically.

## FluentValidation

```bash
dotnet add package Emit.FluentValidation
```

Write a standard FluentValidation validator:

```csharp
public class OrderCreatedValidator : AbstractValidator<OrderCreated>
{
    public OrderCreatedValidator()
    {
        RuleFor(x => x.CustomerId).NotEmpty();
        RuleFor(x => x.Amount).GreaterThan(0);
    }
}
```

Register your validators in DI and wire them into the consumer group:

```csharp
// Register FluentValidation validators
services.AddValidatorsFromAssemblyContaining<OrderCreatedValidator>();

services.AddEmit(emit =>
{
    emit.AddKafka(kafka =>
    {
        kafka.Topic<string, OrderCreated>("orders", topic =>
        {
            topic.ConsumerGroup("order-processors", group =>
            {
                group.ValidateWithFluentValidation(
                    onFailure => onFailure.DeadLetter());
                group.AddConsumer<OrderCreatedHandler>();
            });
        });
    });
});
```

`ValidateWithFluentValidation` resolves `IValidator<TMessage>` from the service provider for each message. If no validator is registered, you get a clear `InvalidOperationException` at runtime.

## Error Actions

Every `Validate` call requires an error action that determines what happens to invalid messages:

| Action | Behavior |
|--------|----------|
| `Discard()` | The message is silently dropped. The consumer never sees it. |
| `DeadLetter()` | The message is forwarded to the configured dead letter topic with diagnostic headers. |

The validation error action is separate from the group-level `OnError` policy. Validation failures use the action you pass to `Validate`; exceptions thrown by your validator (transient errors like a database timeout) fall through to `OnError` like any other exception.

:::note
FluentValidation always runs `ValidateAsync` internally, so both synchronous and asynchronous rules work correctly.
:::
