Middleware in Emit follows the same pattern as ASP.NET: each component receives the context and a `next` delegate. Call `next` to continue the pipeline, skip it to short-circuit.

## Implementing middleware

```csharp
public class TimingMiddleware<T> : IMiddleware<ConsumeContext<T>>
{
    private readonly ILogger<TimingMiddleware<T>> _logger;

    public TimingMiddleware(ILogger<TimingMiddleware<T>> logger)
    {
        _logger = logger;
    }

    public async Task InvokeAsync(ConsumeContext<T> context, IMiddlewarePipeline<ConsumeContext<T>> next)
    {
        var sw = Stopwatch.StartNew();
        try
        {
            await next.InvokeAsync(context);
        }
        finally
        {
            _logger.LogDebug("Message handled in {ElapsedMs}ms", sw.ElapsedMilliseconds);
        }
    }
}
```

For outbound middleware, replace `ConsumeContext<T>` with `SendContext<T>`.

The type parameter `T` is the message type for that specific pipeline level. Emit closes the open generic at registration time for each message type it handles.

## Registering middleware

### Global (all messages, all providers)

```csharp
builder.Services.AddEmit(emit =>
{
    emit.InboundPipeline.Use<TimingMiddleware<>>();   // open generic
    emit.OutboundPipeline.Use<SigningMiddleware<>>();
});
```

### Provider level (all Kafka messages)

```csharp
emit.AddKafka(kafka =>
{
    kafka.InboundPipeline.Use<KafkaHeaderMiddleware<>>();
});
```

### Consumer group level (one group)

```csharp
topic.ConsumerGroup("order-processor", group =>
{
    group.Use<OrderValidationMiddleware<OrderPlaced>>();  // closed generic is fine here
    group.AddConsumer<OrderPlacedConsumer>();
});
```

### Per-consumer level

```csharp
group.AddConsumer<OrderPlacedConsumer>(consumer =>
{
    consumer.Use<ConsumerSpecificMiddleware<OrderPlaced>>();
});
```

## Middleware lifetimes

```csharp
group.Use<MyMiddleware>(MiddlewareLifetime.Singleton);  // one shared instance (default)
group.Use<MyMiddleware>(MiddlewareLifetime.Scoped);     // one instance per message
```

`Singleton` is the default: one instance shared across all concurrent workers. It must be thread-safe and cannot hold per-message state. `Scoped` middleware gets a fresh instance for each message and can safely hold per-message state.

## Accessing services

Middleware is resolved from the DI container. For scoped dependencies, use constructor injection directly (works with `Scoped` lifetime). For singletons that need per-message scoped services, resolve from `context.Services`:

```csharp
public async Task InvokeAsync(ConsumeContext<T> context, IMiddlewarePipeline<ConsumeContext<T>> next)
{
    var repo = context.Services.GetRequiredService<IOrderRepository>();
    // ...
    await next.InvokeAsync(context);
}
```

## Short-circuiting

Skip `next` to abort the pipeline without throwing. The message is acknowledged (offset committed) and not retried:

```csharp
public async Task InvokeAsync(ConsumeContext<T> context, IMiddlewarePipeline<ConsumeContext<T>> next)
{
    if (context.TransportContext is KafkaTransportContext<string> { Key: "__test__" })
    {
        logger.LogDebug("Skipping test message");
        return; // pipeline ends here
    }

    await next.InvokeAsync(context);
}
```

If you want the message to be treated as failed (triggering error policy), throw instead of returning.

## Execution order

Middleware executes in the order it is registered, outermost to innermost. Emit's built-in instrumentation (tracing, metrics, observers) always runs before your global middleware.

The transactional middleware runs inside the retry loop, wrapping only the handler invocation. Custom per-consumer middleware registered at the same level runs outside the transaction unless it is registered after the transactional middleware in the pipeline.
