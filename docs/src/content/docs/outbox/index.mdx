title: Transactional Outbox
description: How Emit's outbox delivers messages reliably without the dual-write problem.
---

The transactional outbox solves the dual-write problem by ensuring that your business data and your message enqueue happen in the same database transaction. Either both commit, or neither does.

## How it works

1. **Atomic write:** When `ProduceAsync` is called within a transaction scope, the outbox entry is written to the database in the same session as your business data. Both commit together.

2. **Background delivery:** A [daemon](../concepts/leader-election/) polls the outbox, picks up pending entries, and delivers them to the broker. Your endpoint returns immediately after the commit.

3. **Node tracking:** Each entry records the `NodeId` of the node that enqueued it, set automatically from `INodeIdentity`. This lets you trace message provenance to the originating node.

4. **Delete on success:** Once delivery succeeds, the entry is deleted. There are no intermediate status fields: entry present means pending, entry absent means delivered.

5. **Automatic retry:** If delivery fails, the entry stays in place and is retried on the next poll cycle. The outbox guarantees at-least-once delivery.

## Choosing your approach

Pick the approach that fits your context:

| You're writing a... | Provider | Recommended approach |
|---|---|---|
| Kafka consumer or mediator handler | EF Core or MongoDB | `[Transactional]` attribute |
| Controller, endpoint, or simple service | EF Core | Implicit (`SaveChangesAsync`) |
| Hosted service or background job | EF Core or MongoDB | `IUnitOfWork` |
| Anything needing conditional commit logic | EF Core or MongoDB | `IUnitOfWork` |

**`[Transactional]` attribute:** the simplest option for consumers and handlers. Add the attribute to your class and write your business logic. The pipeline handles begin, commit, and rollback automatically. Each retry attempt gets a fresh transaction.

```csharp
[Transactional]
public sealed class MyHandler : IRequestHandler<MyCommand>
{
    public async Task HandleAsync(MyCommand request, CancellationToken ct)
    {
        await eventRepository.InsertAsync(request.Event, ct);
        await producer.ProduceAsync(..., ct);
        // Transaction is managed automatically by the middleware.
    }
}
```

**Implicit (EF Core only):** produce messages and call `SaveChangesAsync()`. The outbox entry and your business data share the same change tracker, so they're flushed atomically in a single implicit database transaction. No ceremony needed.

**`IUnitOfWork`:** for scenarios outside the pipeline (hosted services, background jobs) or when you need explicit control over when to commit or rollback. Inject `IUnitOfWork`, call `BeginAsync`, do your work, then `CommitAsync`. Works identically for EF Core and MongoDB. See the [EF Core](../persistence/efcore/) and [MongoDB](../persistence/mongodb/) persistence pages for provider-specific examples.

## Setup

Call `UseOutbox()` on your persistence provider to enable the outbox infrastructure. When `UseOutbox()` is configured, all producers route through the outbox by default. Call `UseDirect()` on any producer that should bypass the outbox.

```csharp
emit.AddMongoDb(mongo =>
{
    mongo.Configure((sp, ctx) => { ... })
         .UseOutbox(options =>
         {
             options.PollingInterval = TimeSpan.FromSeconds(5);  // default
             options.BatchSize = 100;                            // default
         });
});

emit.AddKafka(kafka =>
{
    kafka.ConfigureClient(cfg => cfg.BootstrapServers = "localhost:9092");

    kafka.Topic<string, PizzaOrdered>("pizzas", topic =>
    {
        // ... serializers ...
        topic.Producer();  // routes through outbox by default
    });

    kafka.Topic<string, AnalyticsEvent>("analytics", topic =>
    {
        // ... serializers ...
        topic.Producer(p => p.UseDirect());  // bypasses the outbox
    });
});
```

| Option | Default | Description |
|---|---|---|
| `PollingInterval` | 5 seconds | Minimum time between poll cycles; next poll starts immediately if the previous batch took longer |
| `BatchSize` | 100 | Maximum entries fetched per poll cycle, ordered by sequence (FIFO) |

See [Installation](../getting-started/installation/) for the full setup.

## Ordering guarantees

Messages are ordered by **group key**. The default group key for a Kafka topic is `kafka:{topic}` (e.g., `kafka:pizzas`). All entries within the same group are delivered strictly in sequence.

The worker processes different groups in parallel, so two topics make progress independently. Within a single group, entries are delivered one at a time in enqueue order.

## Single-worker guarantee

The outbox worker runs as a [daemon](../concepts/leader-election/#daemon-coordination): exactly one instance in your cluster processes the outbox at any time. If the assigned node goes down, the leader reassigns the daemon to another node within one heartbeat cycle (default: 15 seconds).

This prevents duplicate delivery from concurrent workers processing the same group.

## Failure behavior

When delivery to the broker fails, the worker logs the error and leaves the entry in place. It will be retried on the next poll. There is no exponential backoff at the outbox level; that is handled at the broker layer (e.g., Kafka producer retries).

The outbox is stubborn persistence, not smart retry logic. It keeps trying until it succeeds. If you need to route messages elsewhere after repeated failures, that belongs at the broker layer. The outbox guarantees at-least-once delivery; everything else is upstream.
